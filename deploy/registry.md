# Plynf Container Registry

Pre-built, multi-arch, signed images for every Plynf service. Pull and run, no source checkout needed.

## Quick reference

```bash
# Pin the version + org you want
export PLYNF_ORG=nico-schindlbeck-jpg            # or 'plynf' once the org exists
export PLYNF_VERSION=v1.7                        # any tag from https://github.com/<org>/plynf/releases

# Pull and start the full 13-service stack
docker compose -f deploy/compose.prod.yml pull
docker compose -f deploy/compose.prod.yml up -d --wait

# Verify everything is healthy
for port in 7421 7422 7423 7424 7425 7426 7427 7428 7429 7430 7431 7432 7433; do
  curl -fsS "http://localhost:${port}/healthz" && echo " ← :${port} ok"
done
```

## Available tags

| Tag pattern | Meaning |
|---|---|
| `:v1.7`, `:v1.7.5`, ... | Semantic release tag. Immutable for the lifetime of the package. |
| `:latest` | Floating tag, updated on every release. Avoid in production. |
| `:sha-<7-char-sha>` | Per-commit tag. Useful for testing specific builds. |
| `@sha256:<digest>` | Immutable content-addressable. Best practice for production pins. |

## Architectures

Every image is built for **both** `linux/amd64` and `linux/arm64`. Docker auto-selects based on your host. On Apple Silicon you get arm64; on x86 servers you get amd64. No manual platform flags needed.

To force a specific architecture for testing:

```bash
docker pull --platform linux/arm64 ghcr.io/<ORG>/plynf-workspace:v1.7
```

## Verifying signatures

Every image is signed with [cosign](https://github.com/sigstore/cosign) using keyless OIDC. The signature ties the image to the GitHub Actions workflow that built it, anchored in Sigstore's Rekor transparency log.

```bash
cosign verify \
  "ghcr.io/<ORG>/plynf-workspace:v1.7" \
  --certificate-oidc-issuer "https://token.actions.githubusercontent.com" \
  --certificate-identity-regexp "^https://github.com/<ORG>/.+"
```

Successful output ends with:
```
[
  {
    "critical": {
      "identity": {"docker-reference": "ghcr.io/<ORG>/plynf-workspace"},
      "image": {"docker-manifest-digest": "sha256:..."},
      "type": "cosign container image signature"
    },
    ...
  }
]
```

A non-zero exit means the image was either not signed by the official pipeline or was tampered with. Treat it as a hard fail in CI/CD.

## SBOM (Software Bill of Materials)

Every release ships with a SPDX-formatted SBOM, generated by [syft](https://github.com/anchore/syft) and attached as a [cosign attestation](https://github.com/sigstore/cosign/blob/main/specs/ATTESTATION_SPEC.md).

To pull and inspect:

```bash
cosign download attestation \
  --predicate-type "https://spdx.dev/Document" \
  "ghcr.io/<ORG>/plynf-workspace:v1.7" \
  | jq -r '.payload | @base64d' | jq '.predicate' > sbom.spdx.json

# Then feed sbom.spdx.json into Trivy / Grype / Snyk / your tool of choice
```

## Pinning by digest (production-grade)

Floating tags like `:latest` or even `:v1.7` are mutable in theory (a `docker push` can overwrite them). For supply-chain-conscious deployments, pin by digest:

```bash
# Resolve the digest for a tag once
docker buildx imagetools inspect ghcr.io/<ORG>/plynf-workspace:v1.7 \
  --format '{{json .Manifest.Digest}}'
# → "sha256:a3f8e5..."

# Then reference @sha256: instead of :tag in compose.prod.yml
```

A helper script (`deploy/pin-compose-digests.sh`, not yet shipped) is on the roadmap to automate this for the full stack.

## Image size targets

| Service | Target | Current |
|---|---:|---:|
| All core services (workspace, gateway, identity, dashboard) | ≤ 90 MB | ~95-110 MB (slim base + uvicorn) |
| MCP servers (8 of them) | ≤ 80 MB | ~75-90 MB |
| Mock MCP | ≤ 60 MB | ~55 MB |

Built on `python:3.11-slim` base. Migration to `python:3.11-alpine` is a v1.8 optimization — depends on whether all C-extension wheels (especially `tiktoken`, `pydantic-core`) ship Alpine variants reliably.

## Updating

When a new release tag is pushed (`v1.8`, `v2.0`, etc.), `release-images.yml` runs automatically. ~12 minutes total for the 13-service × 2-arch matrix to build + sign + SBOM. The smoke-test job verifies all 13 services come up healthy from the freshly-pushed images before marking the release complete.

## When images are not enough

The compose-prod approach assumes Docker is available and the user wants a container stack. If a Plynf user wants something simpler:

- **Embedded mode (Block 6, in progress)**: a single PyInstaller binary that bundles workspace + gateway + identity + dashboard + mock-mcp into one process. No Docker, no images, no orchestration. Trades feature completeness (no OAuth-MCPs) for simplicity.
- **`plynf` CLI (Block 3, in progress)**: a Go binary that wraps `docker compose` underneath, so users say `plynf up` instead of `docker compose -f deploy/compose.prod.yml up -d`. Better UX for non-Docker-natives.

This page documents the production-Compose path. The other two paths will be linked from `INSTALL.md` once they ship.
